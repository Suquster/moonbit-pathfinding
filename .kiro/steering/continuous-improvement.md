# 持续自我优化指令（Continuous Self-Improvement Directive）

本规则要求 AI 助手在每次交互中**主动发现并消除项目中的不足**，直到没有合理的优化空间为止。

---

## 一、核心原则：永不满足

1. **每次完成一个任务后**，必须立即审视：还有什么可以改进的？
2. **不要等用户指出问题** — 主动发现、主动修复、主动拔高
3. **不要说"已经足够好了"** — 除非能证明已达到业界对标库的同等水平
4. **优化是无限循环** — 完成一轮优化后，立即开始下一轮，直到用户说停

---

## 二、每轮优化必须检查的维度（按优先级排序）

### P0 — 必须立即修复
- [ ] 是否还有 O(n²) 字符串拼接（循环内 `out = out + ...`）？
- [ ] 是否有 `abort()` / `todo!()` / `unimplemented` 占位？
- [ ] 是否有字符串模拟结构化数据（如 `op: String` 代替枚举）？
- [ ] 是否有空壳函数（函数体为空或只返回默认值）？
- [ ] 测试是否全部通过（wasm-gc / js / native 三后端）？

### P1 — 本轮应该修复
- [ ] 每个公开函数是否有属性测试？覆盖率如何？
- [ ] 是否有可以提取为共享工具的重复代码？
- [ ] 错误处理是否完备（所有错误路径有 Result/Option 返回）？
- [ ] API 命名是否一致、是否符合 MoonBit 惯例？
- [ ] 文档（README.mbt.md / CHANGELOG.md）是否与实现同步？

### P2 — 下一轮应该优化
- [ ] 性能关键路径是否有基准测试？
- [ ] 是否可以添加更多语言特性（mini-compiler）？
- [ ] 是否可以支持更多协议功能（serialization/lsp）？
- [ ] 是否可以增强诊断/调试能力（logging/dst）？
- [ ] 代码注释是否清晰、是否有过时注释？

### P3 — 长期改进
- [ ] 是否可以添加跨 spec 的集成测试？
- [ ] 是否可以添加性能回归基准？
- [ ] 是否需要重构某些模块的内部结构？
- [ ] 是否有可以参考的最新论文/算法可以引入？

---

## 三、自动化优化循环

每次对话中，按以下循环执行：

```
WHILE 存在优化空间:
    1. 运行 moon test，确认当前基线全部通过
    2. 按 P0 → P1 → P2 → P3 优先级扫描所有 spec
    3. 列出发现的问题（用中文）
    4. 对每个问题实施修复
    5. 运行 moon info && moon fmt && moon test 验证
    6. 如果有变更，提交 commit
    7. 回到步骤 2
```

---

## 四、"没有优化空间"的定义

只有同时满足以下所有条件，才能声明"没有优化空间"：

1. P0 检查清单全部 ✅
2. P1 检查清单全部 ✅
3. P2 检查清单中至少 80% 已完成
4. 每个 spec 的测试覆盖率 > 80%（可用 `moon coverage analyze` 检查）
5. 所有公开函数都有属性测试（≥100 迭代）
6. 与业界对标库的功能差距 < 30%（按 requirements.md 中的功能清单计算）
7. 无任何已知的性能瓶颈未被记录或修复

---

## 五、交互规则

- **全程中文**交互
- 每轮优化开始前，用中文向用户报告：发现了什么问题、计划怎么修
- 每轮优化结束后，用中文向用户报告：修了什么、还剩什么
- 如果用户说"停"或"够了"，立即停止优化循环
- 选择窗口必须标明 🟢🔵🟣 档位和中文名称，默认 🟣 档位3
