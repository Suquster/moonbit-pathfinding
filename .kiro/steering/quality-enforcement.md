# 全局质量强制约束（Quality Enforcement Rules）

本规则对所有 spec 的实现过程施加强制约束，不可绕过、不可降级。

---

## 一、代码质量硬性标准

### 1. 禁止 O(n²) 字符串拼接
- **绝对禁止**在循环或递归中使用 `out = out + s` 或 `result = result + str` 模式
- 所有文本构建必须使用高效方案：`Array[Char]` 缓冲、`Array[String]` + `String::concat` / join、或共享的 `Shared_Text_Builder`
- 单次拼接（非循环）允许使用 `+`，但超过 3 次顺序拼接必须切换为缓冲方案
- **检查方法**：实现完成后，在所有 `.mbt` 源文件（非测试文件）中搜索循环内的字符串 `+` 操作，确保为零

### 2. 禁止占位实现（stub/placeholder）
- **绝对禁止** `abort()`、`todo!()`、`unimplemented`、`panic`、`raise` 作为功能占位
- **绝对禁止**空函数体或只返回默认值的函数作为"先实现再补全"的占位
- 每个公开函数必须有完整的、可工作的实现
- 如果某功能确实无法在当前 MoonBit 版本实现，必须在代码注释和文档中明确标注原因，并提供最佳近似实现

### 3. 禁止硬编码和 magic number
- 所有常量必须命名（`let MAX_DFA_STATES = 10000`，不是直接写 `10000`）
- 算法参数必须可配置，不能写死在函数内部

### 4. 类型安全
- **禁止**使用 `Any` 类型绕过类型检查
- **禁止**使用字符串模拟结构化数据（如 `TargetInstr { op: String }` 必须改为枚举）
- 所有数据结构必须使用精确的类型表示

---

## 二、测试质量硬性标准

### 1. 属性测试（PBT）要求
- 每个新增的公开函数至少有一个属性测试
- 每个属性测试运行 **不少于 100 次迭代**（`holds_for_all` 的 `count` 参数 >= 100）
- 属性测试必须覆盖：
  - 往返一致性（roundtrip）：`decode(encode(x)) == x`
  - 代数定律：结合律、交换律、幂等性（如适用）
  - 等价性：新实现与旧实现对相同输入产出相同结果
  - 边界条件：空输入、最大输入、特殊字符

### 2. 三后端一致性
- 所有测试必须在 `wasm-gc`、`js`、`native` 三个后端通过
- 如果某后端暂时不支持某功能，必须用条件编译或标记跳过，并在文档中说明
- **禁止**只在一个后端测试就认为通过

### 3. 禁止修改现有测试
- **绝对禁止**修改任何现有测试文件来让新代码通过
- 如果新代码导致现有测试失败，说明新代码有 bug，必须修复新代码而不是改测试
- 唯一例外：如果现有测试本身有明确的 bug（如断言值写错），可以修复，但必须在 commit message 中说明

---

## 三、架构约束

### 1. API 冻结策略（freeze + bypass）
- **绝对禁止**修改或删除任何现有公开函数/类型的签名
- 新功能只能通过新增函数/类型的方式添加（bypass）
- `.mbti` 接口文件只允许新增条目，不允许删除或修改现有条目
- 检查方法：`moon info` 后 diff `.mbti` 文件，确认只有新增行

### 2. 向后兼容
- 所有现有的使用方式必须继续工作
- 如果需要改变内部行为，外部接口必须保持不变
- deprecated 的函数放入 `deprecated.mbt` 但不删除

### 3. 包依赖控制
- 新增的共享工具放在合适的基础设施包（如 `infra_pbt` 或新建 `infra_util`）
- **禁止**跨 spec 方向包之间的直接依赖（如 `regex-engine` 不能依赖 `serialization`）
- 共享代码只能放在基础设施包中

---

## 四、实现过程约束

### 1. 每步验证
- 每修改一个文件后，必须运行 `moon test` 确认不破坏现有测试
- 每完成一个方向的加固后，必须运行完整的 `moon info && moon fmt && moon test`
- 如果某步骤导致测试失败，**必须立即修复后再继续**，禁止跳过失败继续前进

### 2. 逐步推进
- **禁止**一次性修改所有 spec — 必须逐个 spec 加固、逐个验证
- 推荐顺序：先实现共享工具（StringBuilder）→ 再按依赖关系从底层到上层逐个加固
- 每完成一个 spec 的加固，做一次 git commit

### 3. 代码风格
- 遵循 MoonBit `///|` 代码块风格
- `moon fmt` 格式化结果为准
- 注释使用中文
- deprecated 代码放入 `deprecated.mbt`

---

## 五、交互语言约束

- **全程使用中文**与用户交流
- spec 文档（requirements.md、design.md、tasks.md）使用中文
- 代码注释使用中文
- 选择窗口、确认提示、进度报告全部使用中文
- 档位标识必须显示中文名称和编号（如 `🟣 档位3 业界顶尖`）
