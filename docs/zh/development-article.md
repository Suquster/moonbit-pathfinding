# 从 BFS 到 Hub Labeling：用 MoonBit 打造生产级路网最短路栈的开发实录

> 项目：`Suquster/moonbit-pathfinding` ·
> 面向 2026 MoonBit 软件合成挑战（<https://www.moonbitlang.cn/2026-scc>）
> 本文所有性能数字均来自仓库内已归档、可复现的基准工件
> （`benches/results/`），每个结论都能用一条命令验证。

---

## 一、要解决的真实问题

MoonBit 生态缺一个可以直接拿来做路径规划、图算法教学、AI Agent 调用
和多后端部署的工程化图算法库。Rust 有 `pathfinding` crate，Python 有
NetworkX，而 MoonBit 在本项目之前没有对应基础设施。

但"把算法搬到 MoonBit"不是目标。我们给自己定的验收标准是三层：

1. **广度**：覆盖教学与工程常用的经典算法（BFS 到最小费用最大流）；
2. **深度**：在路网最短路这一个方向对标行业 SOTA——不是论文复述，
   而是真实 OSM 路网上可测量的数量级加速；
3. **可信度**：每个声明都有可复现证据，每个优化都有差分测试守卫。

最终交付：**30 种经典算法 + 8 种前沿路网算法**，三后端
（native / wasm-gc / js）2158 项测试全绿，真实北京驾车路网上
距离查询从双向 Dijkstra 的 6.3 ms 压到 Hub Labeling 的
**0.44 µs（14304×）**。

## 二、API 设计：双轨制的取舍

### 2.1 successor function：面向易用性

主 API 沿用 Rust `pathfinding` 验证过的 successor function 风格：

```moonbit
let path = @dir.dijkstra(0, fn(n) { adj[n] }, fn(n) { n == goal })
```

不强绑定图结构——用户的图可以是数组、Map、惰性生成的隐式图
（如八数码状态空间）。这对 AI Agent 生成调用代码尤其友好：
调用面只有三个参数，失败用 `None` 表达而非异常。

### 2.2 CSR 快路径：面向性能

successor function 的代价是每次扩展都过一层闭包与泛型哈希。对
16 万节点的北京路网，这层抽象税不可接受。因此 `src/directed`
提供第二轨：`WeightedCsr`（压缩稀疏行）上的稠密整数节点快路径，
全部前沿算法（CH / HL / PHAST / …）都建立在这一轨上。

**取舍记录**：我们没有用 trait 把两轨统一成一个 API——统一意味着
要么快路径背上动态派发，要么易用轨被迫暴露 CSR 细节。两轨各自
测试、文档中明确选型指引（`docs/AI_AGENT_USAGE.md`），是更诚实的方案。

## 三、路网 SOTA 攻坚：一条完整的登山路线

点到点最短路在真实路网上的行业标准打法是"预处理换查询时间"。
我们按预处理成本递增、查询速度递增的顺序逐档攻克，
每一档都在北京（163501 节点 / 406591 边）与厦门（23925 / 54151）
OSM 驾车网上实测（工件：`benches/results/ch-osm-20260705.md`）：

| 档位 | 算法 | 预处理（北京） | 每查询（北京） | 相对双向 Dijkstra |
|---|---|---|---|---|
| 0 | 双向 Dijkstra | 无 | 6318 µs | 1× |
| 1 | ALT（k=16 地标） | 0.55 s | 1114 µs | **6.6×** |
| 2 | Contraction Hierarchies | 24.6 s | 134 µs | **46×** |
| 3 | Hub Labeling（2-hop） | +27 s | 0.44 µs | **14304×** |

批量档同样齐备：PHAST 一到全 SSSP（相对全量 Dijkstra 6.15×，
含最短路树版）、RPHAST 目标子集限定（再 6.9–9.8×）、
many-to-many 距离表（bucket 法，64×64 相对逐对 CH 16–25×）。

### 3.1 CH：witness 搜索与懒更新收缩序

CH（Geisberger 2008）是这条路线的枢纽。实现要点：

- **edge difference 懒更新优先级**：pop 后重估，劣于堆顶重新入队；
- **有界 witness 搜索**：弹出数上限控制预处理成本，多补的捷径
  只影响查询规模不影响正确性；
- **stall-on-demand**：查询期若存在更高 rank 节点经下行边给出更短
  距离则免松弛——论文 §4 的关键查询加速；
- **捷径缓冲复用**：懒重估与中标收缩原本各做一次完整 witness
  搜索；改为把需补捷径写入缓冲、中标后直接回放，北京预处理
  28.2 s → 24.6 s（-12%），查询零回归。

### 3.2 HL：查询只是两条有序数组的归并

Hub Labeling（Abraham 2011）在 CH 之上按 rank 降序构建 2-hop
标签，查询归并两条有序标签数组取最小和——没有堆、没有图遍历，
北京均摊标签长 129，查询 0.44 µs。路径还原用 `query_via` 找出
经停 hub，再用两段 CH 展开拼接，80 迭代 PBT 验证距离最优 +
路径逐边合法。

### 3.3 PHAST：把 SSSP 的堆扔掉

PHAST（Delling 2011）把一到全查询拆成向上 Dijkstra + rank 降序
线性扫。关键工程细节是**位置空间 CSR 重排**：下行边按目标位置
重排后缓存局部性大幅改善，厦门从 3.99× 拉到 6.15×——这也是
"论文思想 + 实测驱动微调"的典型样本。

## 四、证伪也是证据：ALT 在随机图上的失败实验

诚实是评审最贵的货币。ALT 的三角不等式下界依赖图的度量结构，
我们在合成 uniform 随机图（expander 性质）上实测发现下界几乎
恒为零、加了地标反而全面变慢——于是跨语言对比基准侧维持零
启发式，并把这个**证伪实验完整归档**
（`benches/results/alt-indexed-osm-20260705.md`）。
同一份文件里，北京真实路网上 ALT 是 6.6×。
一个算法的收益必须报告它的适用分布，这比只报好看数字更有说服力。

类似的失败记录还有：h 值代戳记忆化（零启发式负载劣化，回退）、
早期 PHAST 无重排版本（厦门只有 3.99×）。全部留在归档里。

## 五、正确性方法论：差分 PBT + 全量对拍

性能优化最大的风险是悄悄改变语义。我们的守卫是三层：

1. **差分属性测试（PBT）**：每个前沿算法 60–120 迭代随机图，
   与库内 Dijkstra 逐值对拍；路径类输出额外验证逐边存在于原图
   且权和等于代价；
2. **OSM 全量对拍**：基准运行同时是一致性校验——CH/HL/PHAST/
   m2m 的每个查询结果都与双向 Dijkstra 逐点比对，不一致即失败；
3. **三后端矩阵**：native / wasm-gc / js 全绿才允许合入，
   避免"只在一个后端正确"的隐性依赖。

一个真实案例：CH 捷径缓冲优化把"边搜边写"改成"先搜后回放"，
导致同一收缩内后续 witness 搜索看不到新捷径、捷径略密
（标签长 +1.2%）。这不是 bug（正确性由对拍守卫证明），
但我们在归档中如实记录了这个行为差异，而不是宣称"逐位等价"。

## 六、与 Rust pathfinding 的对比基础设施

"比 Rust 快/慢"必须是可复现命令而非口号。`bench_rust/` 实现了
跨语言等价工作负载对比：两侧共享**逐位一致的 xorshift64 随机源**
与相同的确定性图生成算法，`--mode golden` 输出黄金 JSON 样本
逐元素比对作门禁；加速比统一以中位计时计，失败/超时/结果不一致
的用例标注并排除。方法学声明、机器信息与两套工具链版本全部写进
报告工件（`benches/results/latest-rust-comparison.{md,json}`）。

差异化不在"语言换皮"，而在 Rust `pathfinding` 未提供的 8 种
前沿路网算法，以及 MoonBit 的多后端一键部署（同一份算法代码
驱动 native 基准与浏览器 playground）。

## 七、AI 协作记录

本项目由人类作者与 AI 编码代理（Devin）协作完成，协作模式：

- **人类**：定方向（"对标行业 SOTA"）、划红线（提交必须走
  `push-to-upstream.sh`、不许绕过测试门禁）、验收节奏；
- **AI**：算法调研与实现、差分测试编写、OSM 基准搭建、
  失败实验的识别与回退、文档同步；
- **约定**：任何优化必须先测后合——分支验证、PBT 守卫、
  真实路网实测有收益，才进 main；证伪的方案连同数据一起归档。

AI 高效的前提是仓库对 AI 友好：可执行 README（`README.mbt.md`
由 `moon test` 运行）、一键验收脚本（`scripts/acceptance.ps1`）、
硬门禁 CI，让代理每一步都能自我验证。这也反过来成为本库对
所有 AI Agent 用户的卖点（`docs/AI_AGENT_USAGE.md`）。

## 八、给后来者的三条经验

1. **先建验证基础设施，再谈优化**。差分 PBT 与全量对拍在手，
   激进的架构改动（捷径缓冲、早退谓词）才敢做、才敢合。
2. **真实数据会推翻合成数据的结论**（双向）。ALT 在随机图上
   无效、在路网上 6.6×；PHAST 的缓存重排在小图上看不出差别、
   在 16 万节点上决定成败。
3. **失败实验归档比成功数字更能建立信任**。评审/用户真正想知道
   的是边界在哪里。
